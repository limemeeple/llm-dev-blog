## 왜 이 블로그를 시작했나

JAVA 개발 경험을 살려서 LLM 개발을 시작해보라는 강사분의 조언을 듣고 경험이 있던 ERP에 AI 기능을 얹어보자는 마음에 인공지능 API를 이용해 ERP 등록 챗봇을 만들던 도중 의문이 들었다.

'과연 단순 인공지능 API를 붙여서 사용하는게 LLM 개발이 맞는건가?'

그래서 강사분을 찾아가 다시 조언을 구했을때 RAG에 대한 내용을 들었다.

## RAG에 대한 키워드를 듣고 ?

강사님께서 말씀하신 내용 중에 기억나는 부분은 아래와 같았다.

- 문서와 데이터를 학습시켜서 답변을 이끌어내는 것
- 임배딩
- 벡터 DB

다 알아듣지는 못했지만 위와 같은 키워드를 전달 받아 RAG 프로젝트를 진행하기 위해 AI에게 질문을 던졌다.

'LLM 프로젝트를 해보고 싶다.'

## '이게 LLM 개발자구나' 싶은 핵심경험 정리

# 1단계 — API 붙여서 무언가 만들기
LLM API(Claude, OpenAI 등)를 호출해서 간단한 챗봇이나 요약기, 분류기를 만들어봐. 여기서 배우는 핵심은 코드가 아니라 프롬프트 설계랑 응답을 어떻게 파싱해서 앱에 녹이는가야. 예를 들면 "이메일을 넣으면 카테고리 분류 + 요약 + 답장 초안을 JSON으로 뱉는 서비스" 같은 거. Spring Boot로 만들면 네 강점이랑 딱 맞아.

# 2단계 — RAG (이게 진짜 핵심이야)

Retrieval-Augmented Generation. 내 문서/데이터를 LLM이 참고해서 답하게 만드는 건데, 현업 LLM 개발에서 가장 많이 하는 작업이야. 흐름은 이래: 문서를 잘라서(chunking) → 임베딩으로 벡터화 → 벡터 DB에 저장 → 질문 들어오면 관련 조각 검색 → LLM한테 같이 던져서 답변 생성.
추천 토이 프로젝트: "내 PDF/노션/사내 문서에 질문하면 답해주는 봇." 이거 하나 제대로 만들면 임베딩, 벡터 DB(예: pgvector, Qdrant), 청킹 전략, 검색 품질 같은 실무 감각이 한 번에 들어와. 처음엔 검색이 엉뚱한 걸 물어오는 경험을 꼭 하게 되는데, 그걸 개선하는 과정이 진짜 실력이 붙는 구간이야.

# 3단계 — 에이전트 / 툴 사용
LLM이 도구(검색, 계산, API 호출 등)를 스스로 골라 쓰면서 여러 단계로 문제를 푸는 거. "질문 받으면 → 웹 검색하고 → 결과 정리해서 답하는" 식. 여기서 멀티스텝 추론, 함수 호출(tool/function calling)을 다루게 돼.

# 4단계 — 평가(evaluation)
의외로 이게 주니어/시니어를 가르는 부분이야. "내 RAG 봇이 잘 작동하는지 어떻게 숫자로 증명할래?" LLM 출력은 비결정적이라 테스트가 까다롭거든. 작은 평가셋 만들어서 정확도/환각 비율 측정하는 하니스를 짜보면, 단순 데모 수준을 넘어서게 돼.

## 첫 프로젝트: 문서 기반 Q&A 시스템

거창한 챗봇 데모가 아니라, "엔지니어링"이 보이는 프로젝트를 만들기로 했다.

| 레이어 | 언어 / 기술 | 역할 |
| --- | --- | --- |
| API · 서비스 | Java (Spring Boot, Spring AI) | 요청 라우팅, LLM 호출 오케스트레이션 |
| 데이터 · 평가 | Python | 청킹 · 임베딩 파이프라인, 평가 스크립트 |
| 저장소 | 벡터 DB (예정) | 임베딩 검색 |

핵심은 단순히 "작동하는 RAG"에서 멈추지 않고,
**검색 품질 개선의 before/after**와 **정량 평가**까지 가는 것이다.
